# Git 컨벤션, 규칙 관리

> 💡 Git 관련 컨벤션, 규칙을 정리한 문서입니다. 궁금한 점이 있으면 언제든 편하게 말씀해주세요.

# 소개

원활한 코드 관리를 위해 Git 컨벤션을 정리하고자 합니다! 부족한 부분도 많고, 개선해야 할 부분이 있을 것으로 생각하는데, 적용해보시면서 좋은 점, 나쁜 점 피드백 주시면 논의를 통해 적극 반영하겠습니다.

# 커밋 관리

## 규칙

### 1. 커밋 메시지는 영문으로 작성

커밋 메시지는 영문으로 작성해주세요. 한글도 좋고 영문도 좋지만, 하나로 통일하여 가독성을 지키고자 합니다.

### 2. 커밋 메시지를 자세히 작성

커밋 메시지만 보고도 어떤 작업이 이뤄졌는지 대략적으로 이해할 수 있어야 합니다. 커밋 메시지를 작성한 후에, 내가 추가로 전달할 수 있는 정보가 있는지 10초 정도만 진지하게 고민하는 시간을 가져보면 충분할 것 같습니다.

말로만으로는 이해가 조금 어려울 수 있는데, 어떤 생각의 단계를 통해 커밋 메시지를 작성했는지 유추해 볼 수 있도록 작성해보았습니다. 예시이니 참고로만 봐주시면 좋을 것 같습니다.

- 1단계: **오류 수정** → 무슨 오류를 수정한거지?
- 2단계: **상품 삭제 시** (발생하는) 오류 수정 → 왜 오류가 발생하는거지?
- 3단계: 상품 삭제 시 **연결된 주문 내역 유무 잘못 체크하는** 오류 수정

### 3. 자주 커밋하기

커밋은 적당한 작업 단위에서 끊어주는 편이 좋습니다. 작게 자주하는 커밋한 이력은 따로 설명하지 않아도 커밋 메시지만으로 어떤 작업을 했는지 히스토리를 파악할 수 있습니다.

### 4. Gitmoji 사용 (참고사항)

커밋 가독성을 위해 깃모지를 사용하고자 합니다! (필수 아닙니다! 참고만)

깃모지가 무엇인지에 관한 설명은 아래 링크를 참고하시면 좋을 것 같습니다.

- https://gitmoji.dev/
- https://inpa.tistory.com/entry/GIT-⚡️-Gitmoji-사용법-Gitmoji-cli

### 4. 이모지는 정해진 것 안에서 사용

깃모지에서 제공하는 이모지가 너무 많다보니 모든 종류를 다 사용하게 되면 오히려 혼란스럽습니다. 우리에게 꼭 필요한 것 일부만 사용하고자 어떤 작업을 어떤 이모지로 표현할지 간단하게 정리하였습니다.

- 신규 기능: ✨
- 리팩토링: ♻️
- 성능 개선: ⚡️
- 문서화: 📝
- 오류 수정: 🐛
- 핫픽스: 🚑️
- DB 작업(마이그레이션 등): 🗃️
- 코드/파일 삭제: 🔥
- 코드/파일 이동(이름 변경): 🚚
- 로그 추가: 🔊
- 설정 변경: 🔧
- 위에 해당하지 않는 이모지는…
  - 코드 구조 개선에 사용되는 🎨는 ♻️로 표시
  - 오타 수정에 사용되는 ✏️ 는 🐛로 표시
  - 주석 추가에 사용되는 💡는 📝로 표시

### 5. 이슈에 대한 커밋 시 이슈번호 명시

커밋 메시지 마지막에 #이슈번호 와 같은 형태로 적어주면 자동으로 해당 커밋이 이슈에 트래킹됩니다.

## Q&A

- Q. 이모지 하나로 커밋 설명이 어려워요.
  > A. 커밋 단위가 잘못되었을 가능성이 있습니다. 더 작은 단위로 나누어 커밋해봅시다.
- Q. 위의 사용 중인 이모지가 제가 작업한 내용을 잘 설명하지 못하는 것 같아요.
  > A. 깃모지에서 사용 가능한 이모지의 종류가 많다보니 필요한 것 몇가지만을 위주로 가져왔습니다. 분리가 필요하다거나 새로 도입이 필요하다면, 적극적으로 의견을 제안해주세요.

# 브랜치 관리(전략)

## 규칙

### 1. 모든 신규 브랜치는 main 브랜치에서 분기

모든 브랜치는 main 브랜치에서 분기합니다.

### 2. 신규 브랜치는 api/이름, client/이름, agent/이름, common/이름 을 사용

일반적으로는 서로 다른 레포지토리에 있어야 할 소스코드를 하나로 구성하였기 때문에, feature 브랜치 대신 각 작업 도메인 별로 브랜치명을 사용하고자 합니다. 각자 본인 작업에 맞는 브랜치 이름으로 만들어주시고, 여러 도메인에 걸쳐서 작업이 필요하다면 common/이름 형태로 사용해주세요.

### 3. main 브랜치에 충돌 발생 시 로컬 feature 브랜치에서 충돌 해결

main 충돌이 발생하더라도 수정은, 로컬 feature 브랜치에서 이루어져야 합니다. main 브랜치에 직접 커밋하는 일이 없도록 해주세요! 또한, 충돌 해결할 때에는 병합하기 전에 꼭 빌드가 잘되는지 확인 부탁드립니다.

### 4. 충돌은 병합하고자 하는 사람이 기존 코드 작성자와 조율하여 해결

충돌이 발생했을 때에는 병합하고자 하는 사람이 기존 코드 작성자들과 조율하여 해결합니다. 간단한 수정이라면 해결 후에 공유해도 좋지만, 확신이 없을 때에는 꼭 이전 작업자와 논의합니다.

### 5. 병합 후 특이사항 없는 브랜치는 삭제

main 브랜치에 병합이 끝나고 특이사항이 없는 브랜치는 방치하지 말고 삭제해주세요.

### 6. 브랜치 이름은 소문자와 슬래시(/), 대시(-)만 사용

크로스 플랫폼 호환성을 위해 대소문자 혼용하지 않고, 소문자와 슬래시(/), 대시(-)만 사용해주세요.

# PR 관리

## 규칙

### 1. PR은 공유하고 승인 받은 이후에 병합

PR은 꼭 팀원과 공유하고 승인 받을 수 있도록 해주세요! 크로스 체크가 기본이 되었으면 좋겠습니다. 작업 일정 상 팀원의 확인이 늦어진다면 스스로 한번더 점검하고 병합해주세요.

### 2. PR 내용을 자세히 작성

PR 템플릿을 통해 어떤 작업을 했는지가 드러나게 적어주세요!

### 3. 남이 공유해준 PR은 꼼꼼히 확인

PR 승인은 꼼꼼하게 처리해주세요. 좋은 코드라면 칭찬해주시고, 의문점이 있으면 코멘트를 많이많이 남겨주세요.

### 4. 작업하고자 하는 사항은 issue 발행

작업 사항을 issue로 발행하여 처리하면 현재 어떤 작업을 진행 중이고, 어떤 작업이 완료되었는지 서로 쉽게 확인이 가능합니다. 어렵지는 않아서 이번 기회 통해 한번 활용해보시면 좋겠습니다. (필수 x)
